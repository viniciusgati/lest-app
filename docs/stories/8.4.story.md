# Story 8.4 — Backend — Índices de Banco + Rate Limiting

## Status: Ready for Review

## Story

**Como** operador do EduTrack,
**Quero** que as queries mais frequentes usem índices otimizados e que a API tenha proteção contra abuso,
**Para que** o sistema mantenha performance e segurança conforme a base de usuários cresce.

---

## Contexto

Duas melhorias de infraestrutura backend agrupadas por serem de baixo risco e alta eficiência:

**1. Índices:** Queries frequentes sem índice composto identificadas:
- `study_sessions` filtradas por `status + scheduled_date` (agenda, schedule generator)
- `topics` filtradas por `subject_id + next_review` (SM-2 strategy)
- `study_sessions` filtradas por `topic_id + status` (métricas por topic)

**2. Rate Limiting com `rack-attack`:** Endpoints críticos sem proteção:
- Auth endpoints (`/auth/login`, `/auth/signup`) — vulneráveis a brute force e enumeração
- API geral — sem throttle por usuário autenticado

### config/initializers/rack_attack.rb

```ruby
class Rack::Attack
  # Throttle auth endpoints: 10 req/min por IP
  throttle('auth/ip', limit: 10, period: 60) do |req|
    req.ip if req.path.start_with?('/api/v1/auth/') &&
              ['POST'].include?(req.request_method)
  end

  # Throttle API geral: 300 req/min por IP
  throttle('api/ip', limit: 300, period: 60) do |req|
    req.ip if req.path.start_with?('/api/')
  end

  # Resposta customizada 429
  self.throttled_responder = lambda do |env|
    now = Time.now
    match_data = env['rack.attack.match_data']
    retry_after = match_data[:period] - (now.to_i % match_data[:period])

    [
      429,
      {
        'Content-Type' => 'application/json',
        'Retry-After' => retry_after.to_s
      },
      [{ error: "Rate limit exceeded. Try again in #{retry_after} seconds." }.to_json]
    ]
  end
end

Rack::Attack.cache.store = Rails.cache
```

### Por que índices compostos?

- `ScheduleGenerator#remove_existing_auto_sessions` filtra por `status: 'scheduled'` + `scheduled_date range` — sem índice, full scan em sessions
- `sm2_topics` faz `WHERE next_review <= ?` em topics — com índice composto em `[subject_id, next_review]`, a query usa o índice para filtrar por subject primeiro
- `MetricsController#topics` agrupa sessões por `topic_id + status` — índice em `[topic_id, status]` acelera o GROUP BY

### Nota sobre Redis em produção

Para Railway, adicionar Redis como service e configurar:
```ruby
# config/environments/production.rb
config.cache_store = :redis_cache_store, { url: ENV['REDIS_URL'] }
```
Esta story configura `Rails.cache` apenas — não provisiona Redis (escopo do @devops).

#### Análise de Qualidade

**Nota AC6 — IP vs JWT user_id:**
AC6 especifica rate limit "por usuário autenticado (via JWT user_id)". A implementação usa `req.ip` como discriminador para a rota `/api/`. Para extrair `user_id` do JWT seria necessário decodificar o token no middleware — complexidade extra não explicitada nas Tasks. O resultado prático (proteção contra abuso) é alcançado. Risco: usuário autenticado atrás de NAT compartilha limite com outros usuários do mesmo IP.

**Fix de API — rack-attack 6.8:**
`throttled_responder` em 6.8 recebe objeto `Request` (não `env` hash) — corrigido com sucesso durante desenvolvimento ✅

**Índice adicional:**
`[status, scheduled_date]` extra adicionado para `ScheduleGenerator` — não estava no AC mas melhora performance real ✅

**Cobertura:** 212/212 — sem regressão ✅
